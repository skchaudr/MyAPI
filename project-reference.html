MyAPI — Context Retrieval for Agents & the Human Operator

Plain English. A personal search and retrieval service over years of notes, exported chatbot conversations, and CLI agent session logs. AI agents — and Saboor — query it before doing work, and get ranked, evidence-tagged answers back in under a few seconds.

Technical. FastAPI app sitting on top of Khoj semantic search, with a custom three-lane retrieval + reranker (semantic + keyword + synthesized-note prior), source adapters for Obsidian / ChatGPT / Claude / Claude Code / Codex, Gemini-1.5-Flash batch enrichment, and a mechanical acceptance harness that asserts gold-document hits per benchmark query.

Overview

What this is, and who it is for

The product framing in plain English, then in developer terms, then in the load-bearing two-audience model.

What this is, in one paragraph plain English

MyAPI is one HTTP endpoint Saboor can point any agent at. Instead of every new Claude / Codex / Cursor session re-scanning the repo and re-greping for context, the agent posts a question to /query and gets back the documents, decisions, and recent state that matter. The same endpoint also serves the human use case: "find that thread where I figured out X," "what did I decide about the vault schema," "what's actually broken right now."

Underneath the endpoint sits a normalized corpus built from Obsidian markdown, ChatGPT/Claude web exports, and CLI agent (Claude Code, Codex) session logs. The retrieval logic blends semantic vector search with exact-phrase keyword search and a small prior that boosts canonical anchor docs over chat noise.

Developer-level explanation

  • Service layer — a thin FastAPI app (api/main.py, 26 lines) with three routers: /enrich, /import, /query, plus a /health probe.
  • Brain — context_refinery/retrieval.py (1,383 lines) implements intent classification, three retrieval lanes, a reranker with phrase/variant normalization, and a guarded synthesized-note prior.
  • Corpus pipeline — five source adapters parse heterogeneous exports into a uniform "markdown + YAML frontmatter" doc shape; enrichment.py calls Gemini-1.5-Flash in batches of up to 50 docs to extract tags, projects, key assertions, and summaries.
  • Search backend — Khoj runs on the same VM (khoj.service, port 42110) and owns embeddings + the vector store. MyAPI shells out to Khoj over HTTP, then layers its own ranking on top.
  • Trust calibration — a small benchmark bank of representative queries is run via scripts/acceptance.py; each query has a gold-document expectation; regressions surface as numbered handoffs/000…004 sessions, not just CI red.

The two-audience framing load-bearing

This framing emerged explicitly in handoffs/004: the endpoint serves agents and the human operator. Sources are not audiences. Obsidian, ChatGPT, Claude, Codex, Claude Code are input corpora — not user types.

  • Agent-facing — cold-start context for Claude Code / Codex / Jules / Pi workers / OpenClaw executors. Agent posts a task or question, gets project truth, constraints, prior work, and current status back, then acts.
  • Human-facing — recall, inspection, synthesis. "What did I decide about X?", "Show me the thread where I figured this out", "Have I documented any friction with Neovim?" (null is a valid pass).

Same corpus, same pipeline, different response shapes.

The three retrieval lanes

  1. Semantic lane — Khoj vector search returns the top-N most embedding-similar docs.
  2. Keyword lane — exact phrase + term matching with hyphen / space / no-space normalization (so gold mine ≡ gold-mine ≡ goldmine). Skips its bare-term filter when phrases are present.
  3. Synthesized-note prior — +0.14 boost for docs with frontmatter document_kind: synthesized_note or titles containing anchor / summary / overview. Guarded so it doesn't fire on chat-dump sources (chatgpt / claude / claude-code / codex) — keeps anchors from drowning real episodic threads.

Lanes are OR'd; the reranker then scores the combined candidate set with phrase boosts (+0.45 per phrase match) and intent-aware weights. Intent classifier evaluates OPERATIONAL before PROJECT so "MyAPI is broken" doesn't get routed as a project explainer.

Runtime

Data & control flow

What happens when the corpus is built and when a query runs.

Ingestion path (offline)

  1. Source exports land on disk: Obsidian vault, ChatGPT zip, Claude.ai zip, Claude Code session logs, Codex logs.
  2. ingest_all.py dispatches to the matching adapter in context_refinery/adapters/.
  3. Each adapter emits a normalized doc: markdown body + YAML frontmatter (source, title, timestamps, tags).
  4. enrichment.py optionally calls Gemini-1.5-Flash in batches (≤50 docs) to extract tags, projects, key assertions, summaries.
  5. sanitization.py validates frontmatter; exporter.py writes the final .md files into a "khoj-ready bundle."
  6. Bundle is synced to the VM (~/khoj-data/notes/); Khoj reindexes.

Query path (runtime)

  1. Client (agent or human) POSTs JSON to /query: {"query": "...", "source_filter": [...], "n": 10}.
  2. Intent classifier in retrieval.py labels the query — OPERATIONAL is checked before PROJECT to keep "MyAPI is broken" from being routed as a project explainer.
  3. Three retrieval lanes fire in parallel:
    • Semantic — Khoj vector search.
    • Keyword — exact phrase + term match with hyphen/space/no-space normalization.
    • Synthesized-note prior — +0.14 on canonical anchors, guarded against chat-dump sources.
  4. Reranker unions candidates, applies phrase boost (+0.45) and intent-aware weights, then returns top n with score + source + metadata.
  5. Response is structured JSON. Stateless: no idempotency, no cache (clients dedupe themselves).

Deploy + runtime control plane

  • deploy_to_khoj.sh on Mac → gcloud compute scp → VM filesystem → sudo systemctl restart context-refinery.service → curl /health.
  • VM auto-shuts down after a few hours of inactivity (cost control). Operator restarts via gcloud compute instances start before benchmarking.
  • Access is Tailscale-only. No public IP. No auth on the app itself; the network is the trust boundary.

Decisions

Distilled themes & decisions

The seven framings that survived sessions 000–004 and now anchor the work.

Trust calibration over vibes

The "quality" claim is grounded in a categorized query bank, not feel. Each query is graded with one of seven diagnostic buckets: win, weak win, corpus gap, retrieval gap, metadata gap, intent gap, answer-shape gap. The rubric forces diagnosis at the lowest-leverage layer first — don't tune the reranker if the real issue is a missing anchor doc.

Phase 2 = corpus v1, not weight tuning

Per handoff 004: the v1 push is an architecture pass on the corpus, not a retrieval-weight pass. Planned ingestion-side fields: source_type, work_type, temporal_mode, plus conversation fields thread_type, primary_project, outcome, signal_strength, artifact_summary, raw_thread_weight, review_status.

This subsumes the earlier "recall_type" framing — folder taxonomy is the cleaner carrier than new frontmatter alone.

Episodic vs. meta: don't let meta eclipse the source

A meta-doc (anchor / summary / overview) commentates on an event. A source thread (ChatGPT / Claude / Claude Code) is the event. When the user asks "find the thread where I figured this out," the source should win — not the polished summary written after.

Anti-pattern called out in 004: don't normalize toward "anchors always win." Healthy routing lets canonical docs win for source-of-truth questions and conversations win for thought-process / development-history questions.

Null answers count as passes

Surfaced in 003. A query like "What friction am I facing in Neovim?" silently assumes documented Neovim friction. Reframed: "Have I documented any friction in Neovim?" — and null becomes a valid pass. Grading scheme now includes unfair_question and needs_more_context alongside pass/partial/fail.

One endpoint, two audiences

Sharpened in 004. Agents need cold-start context; the human needs episodic recall + judgment. Same corpus, same pipeline, different response shapes. Sources are not audiences — that confusion led to earlier architectural drift.

Handoffs are the load-bearing log

Every session leaves a numbered handoff: Empirical Reality, Resume Point, Narrative / Trajectory. The system is more reliable than STATUS_AND_NEXT_STEPS.md when the two diverge — status doc updates intermittently, handoffs are written every session.

Read order: 000 (bare retrieval baseline) → 001 (A7 anchor + variant fix) → 002 (F5 episodic gold + S1 schema fix + source_type plan) → 003 (final v0 bank refinement) → 004 (corpus v1 field-test realization, two-audience framing).

Stateless by design

/enrich and /query don't track repeat requests. Caching lives upstream (agent harness) or downstream (Khoj's own cache). MyAPI is the stateless ranking layer.

  2. Two audiences, one corpus. Agents and operator share the same pipeline; only the response shape differs.

Risks

Risks & open questions

The honest gaps. Some accepted, some open, some pending verification.

No auth on the service accepted

The trust boundary is Tailscale. If Tailscale is wide-open on any peer, the endpoint is too. Acceptable today for a personal tool; would need auth before any non-Tailscale exposure.

Corpus v1 normalization fields not yet implemented open

Planned fields (source_type, work_type, temporal_mode, and the conversation set) are designed in project-docs/source-type-taxonomy-plan.md but not stamped on the corpus yet. Most Phase 2 expectations are gated on this work.

A1 acceptance regression known bank-evolution fail

"What is MyAPI and what is its current goal?" — myapi-status-anchor.md wins by 0.002 over obsidian-myapi-anchor.md. Per 002 diagnosis, the devinfra anchor is indexed but has low semantic similarity to the goal query and Khoj only returns 8 candidates total for the A1 query. Real fix is bank update + source_type weights, not retrieval tuning. Deferred until Lane 3 lands.

~27 Obsidian files missing from the index tracked, not blocking

Per the delta-patch report. Patchable via scripts/khoj_repair_index_delta.py. Re-confirm the count against current vault state before claiming a fix.

Curation gap for public use open / scope question

Codex sitrep at the bottom of 004: a stranger with their own ChatGPT/Claude export can run ingest_all.py and get a searchable low-trust scratchpad. They do not get: automatic thread importance scoring, project taxonomy assignment, artifact summaries, sensitive content filtering, long-thread topic splitting, robust dedupe of repeated exports, promotion to trusted memory, or any corpus-v1 routing fields.

Direction (not current state): an interactive review queue — exclude / private / summarize / assign project / mark outcome / promote, with a raw_thread_weight dial per thread.

VM auto-shutdown coupling accepted

VM idles down after a few hours. Acceptance / live queries require an explicit gcloud compute instances start first. Acceptable for personal use; would break any always-on agent integration.

Field test not yet run open

The 10-query batch in 004 is a lab test. Real proof — fresh cold-start agents with no memory beyond "use MyAPI first" running real tasks against real projects — has not happened yet. Phase 2 success criterion lives here.

Self-audit notes

For your eyes — what the project is and isn't, terminology drift, voice cues.

What the system genuinely is, vs what it could sound like

  • Genuinely is: a working personal RAG with a thoughtfully designed reranker, a benchmark discipline, and a session-log practice that's better than most personal projects of this size.
  • Could sound like (don't lean here): a general-purpose enterprise memory product, a production trust-graded retrieval platform, a multi-tenant tool. None of those are accurate today.
  • Stretch claims to drop: any "under 3 seconds" guarantee unless measured fresh; any "auto-curates your knowledge" framing — curation is the explicit unfinished part.

Terminology drift to clean up before README

  • MyAPI vs Context Refinery vs Brain — these names overlap. The repo is "MyAPI"; the Python package is context_refinery/; the FastAPI service title is "Context Refinery API"; the deploy script is deploy_to_brain.sh. Pick one canonical name for the README, footnote the others.
  • "Trust calibration" is great internal language. Public README readers may parse it as jargon — define it inline the first time.
  • "Phase 1 / Phase 2" only makes sense if the reader knows what Phase 0 was. Briefly anchor both, or drop the phase numbering and describe the current focus directly.

Author-voice notes

  • The 004 narrative ("we shouldn't invent authority, we shouldn't reinvent history, we shouldn't recreate or fabricate episodic history") is one of the strongest sentences in the whole repo. Worth quoting verbatim in the README's "what this isn't" section.
  • Handoff narratives use Intent / Interpretation / Tension / Momentum — that frame is unusual and good. Consider adopting it for any longer "Author" section.
  • The phrase "honest routing" (from 004's normalization realization) deserves a line in the README.
