## 2026-04-12 - Fix eval command injection in deploy script
**Vulnerability:** The deployment script `deploy_to_brain.sh` contained an unsafe `eval` statement designed to expand the tilde (`~`) character when evaluating user input for a directory path: `eval LOCAL_DIR="$LOCAL_DIR"`. This introduced a critical command injection vulnerability. A malicious actor could provide input like `"; ls -al; echo "` to execute arbitrary commands with the privileges of the script user.
**Learning:** Shell scripts processing user input should avoid the `eval` builtin wherever possible as it evaluates arbitrary code. While `eval` is often tempting for tasks like tilde expansion, safer alternatives exist in bash.
**Prevention:** Rather than utilizing `eval`, use safe bash parameter expansion constructs. In this case, `LOCAL_DIR="${LOCAL_DIR/#\~/$HOME}"` performs a simple pattern substitution, replacing a leading tilde with the user's home directory path without executing the input as a command.

## $(date +%Y-%m-%d) - Path Traversal Vulnerabilities via `os.path.commonpath`
**Vulnerability:** In Python APIs, parameters that represent optional base paths or directories could be left unexpanded or could resolve to unexpected values when `None`.
**Learning:** `os.path.commonpath` should strictly evaluate expanded strings. In endpoints expecting optional paths (`request.root: Optional[str]`), falling back properly via `(request.root or default_base_path)` ensures the `TypeError` does not occur when `expanduser` is run on a `None` value. Additionally, custom path inputs should be carefully checked to ensure they do not cause functional regressions when validating them against hardcoded internal directories, unless specifically required.
**Prevention:** When validating optional user inputs using `os.path.commonpath`, strictly use `os.path.abspath(os.path.expanduser(request.root or default_path))` to avoid TypeErrors if the parameter defaults to None. Never pass unfiltered user input directly to file system operations.
