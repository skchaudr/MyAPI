## 2026-04-12 - Fix eval command injection in deploy script
**Vulnerability:** The deployment script `deploy_to_brain.sh` contained an unsafe `eval` statement designed to expand the tilde (`~`) character when evaluating user input for a directory path: `eval LOCAL_DIR="$LOCAL_DIR"`. This introduced a critical command injection vulnerability. A malicious actor could provide input like `"; ls -al; echo "` to execute arbitrary commands with the privileges of the script user.
**Learning:** Shell scripts processing user input should avoid the `eval` builtin wherever possible as it evaluates arbitrary code. While `eval` is often tempting for tasks like tilde expansion, safer alternatives exist in bash.
**Prevention:** Rather than utilizing `eval`, use safe bash parameter expansion constructs. In this case, `LOCAL_DIR="${LOCAL_DIR/#\~/$HOME}"` performs a simple pattern substitution, replacing a leading tilde with the user's home directory path without executing the input as a command.

## 2026-04-13 - Prevent Path Traversal in API Endpoints
**Vulnerability:** The API endpoints `import_codex` and `import_claude_code` in `api/routers/imports.py` allowed users to specify a `root` directory for scanning sessions without strictly validating that the provided path resided within the intended base directory. This could potentially allow path traversal attacks where arbitrary directories could be read.
**Learning:** Checking whether a path starts with another using simple string operations like `.startswith()` can introduce a partial path traversal vulnerability. Standardizing paths correctly and enforcing directory boundaries securely is crucial when handling user-provided file paths.
**Prevention:** Always strictly validate user-supplied path parameters by expanding and resolving their absolute paths (`os.path.abspath(os.path.expanduser(path))`) and checking if they share the correct base path using `os.path.commonpath([base_dir, target_path]) == base_dir`. Never pass unfiltered user input directly to file system operations.
