## 2026-04-12 - Fix eval command injection in deploy script
**Vulnerability:** The deployment script `deploy_to_brain.sh` contained an unsafe `eval` statement designed to expand the tilde (`~`) character when evaluating user input for a directory path: `eval LOCAL_DIR="$LOCAL_DIR"`. This introduced a critical command injection vulnerability. A malicious actor could provide input like `"; ls -al; echo "` to execute arbitrary commands with the privileges of the script user.
**Learning:** Shell scripts processing user input should avoid the `eval` builtin wherever possible as it evaluates arbitrary code. While `eval` is often tempting for tasks like tilde expansion, safer alternatives exist in bash.
**Prevention:** Rather than utilizing `eval`, use safe bash parameter expansion constructs. In this case, `LOCAL_DIR="${LOCAL_DIR/#\~/$HOME}"` performs a simple pattern substitution, replacing a leading tilde with the user's home directory path without executing the input as a command.

## 2026-04-13 - Fix path traversal and error leakage in import endpoints
**Vulnerability:** The API endpoints processing import paths (`import_codex` and `import_claude_code`) passed user-provided root paths directly into file-system scan functions without strictly verifying that they stayed within their intended base directories. Moreover, the generic exception handling blocks broadly leaked internal `str(e)` details directly to clients via HTTP 500 errors.
**Learning:** Using basic substring checks like `.startswith()` or failing to sanitize user paths introduces partial path traversal vulnerabilities. Exposing raw error stack traces gives attackers insight into the application's internal structure or specific file paths.
**Prevention:** Always use `os.path.abspath(os.path.expanduser(path))` along with `os.path.commonpath([base_dir, target_path]) == base_dir` to ensure strict path containment. For error handling, use generic safe error messages for clients (`An internal server error occurred`) while logging detailed stack traces (`logger.error(..., exc_info=True)`) server-side.
