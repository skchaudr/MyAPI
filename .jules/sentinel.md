## 2026-04-12 - Fix eval command injection in deploy script
**Vulnerability:** The deployment script `deploy_to_brain.sh` contained an unsafe `eval` statement designed to expand the tilde (`~`) character when evaluating user input for a directory path: `eval LOCAL_DIR="$LOCAL_DIR"`. This introduced a critical command injection vulnerability. A malicious actor could provide input like `"; ls -al; echo "` to execute arbitrary commands with the privileges of the script user.
**Learning:** Shell scripts processing user input should avoid the `eval` builtin wherever possible as it evaluates arbitrary code. While `eval` is often tempting for tasks like tilde expansion, safer alternatives exist in bash.
**Prevention:** Rather than utilizing `eval`, use safe bash parameter expansion constructs. In this case, `LOCAL_DIR="${LOCAL_DIR/#\~/$HOME}"` performs a simple pattern substitution, replacing a leading tilde with the user's home directory path without executing the input as a command.

## 2024-05-24 - [Path Traversal in Import Endpoints]
**Vulnerability:** Import endpoints (`/codex` and `/claude-code`) accepted a `root` parameter without fully validating that the resolved path remained under the expected base directory. This allowed partial path traversal (e.g., escaping the base directory via `../`) when using `os.path.expanduser`.
**Learning:** `os.path.expanduser` alone is insufficient to prevent path traversal because it does not resolve relative path segments like `../` into an absolute boundary check. Simple string matching with `.startswith()` is also vulnerable to partial directory matching.
**Prevention:** To prevent path traversal, user-supplied paths must be securely validated by expanding them, converting to absolute paths (`os.path.abspath(os.path.expanduser(path))`), and then ensuring they share the correct base path using `os.path.commonpath([base_dir, target_path]) == base_dir`. Exception handling must also be generic (e.g. returning 500) so as not to leak path structures in error messages.
