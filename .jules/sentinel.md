## 2026-04-12 - Fix eval command injection in deploy script
**Vulnerability:** The deployment script `deploy_to_brain.sh` contained an unsafe `eval` statement designed to expand the tilde (`~`) character when evaluating user input for a directory path: `eval LOCAL_DIR="$LOCAL_DIR"`. This introduced a critical command injection vulnerability. A malicious actor could provide input like `"; ls -al; echo "` to execute arbitrary commands with the privileges of the script user.
**Learning:** Shell scripts processing user input should avoid the `eval` builtin wherever possible as it evaluates arbitrary code. While `eval` is often tempting for tasks like tilde expansion, safer alternatives exist in bash.
**Prevention:** Rather than utilizing `eval`, use safe bash parameter expansion constructs. In this case, `LOCAL_DIR="${LOCAL_DIR/#\~/$HOME}"` performs a simple pattern substitution, replacing a leading tilde with the user's home directory path without executing the input as a command.
## 2026-05-24 - Fix Path Traversal and Information Leakage
**Vulnerability:** Path traversal was possible in `/import/codex` and `/import/claude-code` endpoints. The `root` parameter provided by the user was not adequately checked against the allowed base directory. Additionally, raw exception strings were returned in 500 responses across multiple API endpoints, leaking internal details.
**Learning:** Python `os.path.abspath(os.path.expanduser())` is needed to safely resolve the exact location and compare using `os.path.commonpath()` to restrict traversal to intended base directories, avoiding risks from users traversing back (e.g. `../../`). For exception handling, sending the literal `str(e)` payload directly to external clients is unsafe.
**Prevention:** Always validate resolved paths to enforce strict root directory constraints for file operations based on external inputs. For error handlers, log errors securely via `exc_info=True` and only ever send a generic "An internal server error occurred" message for HTTP 500 responses.
