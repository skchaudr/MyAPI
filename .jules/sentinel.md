## 2026-04-12 - Fix eval command injection in deploy script
**Vulnerability:** The deployment script `deploy_to_brain.sh` contained an unsafe `eval` statement designed to expand the tilde (`~`) character when evaluating user input for a directory path: `eval LOCAL_DIR="$LOCAL_DIR"`. This introduced a critical command injection vulnerability. A malicious actor could provide input like `"; ls -al; echo "` to execute arbitrary commands with the privileges of the script user.
**Learning:** Shell scripts processing user input should avoid the `eval` builtin wherever possible as it evaluates arbitrary code. While `eval` is often tempting for tasks like tilde expansion, safer alternatives exist in bash.
**Prevention:** Rather than utilizing `eval`, use safe bash parameter expansion constructs. In this case, `LOCAL_DIR="${LOCAL_DIR/#\~/$HOME}"` performs a simple pattern substitution, replacing a leading tilde with the user's home directory path without executing the input as a command.

## 2025-02-27 - Fix path traversal and info leakage in imports router
**Vulnerability:** The `/codex` and `/claude-code` endpoints in `api/routers/imports.py` accepted an arbitrary `root` path string without validation, passing it to `os.walk` in parser functions, creating a path traversal vulnerability. In addition, `except Exception as e` blocks returned raw exception strings `str(e)` in 500 error HTTPExceptions, leaking sensitive internal trace or path information.
**Learning:** Even internal tool endpoints dealing with "roots" or "paths" must resolve and validate against their intended base directories. Any raw `Exception` should be logged server-side and shielded from the client to prevent information leakage.
**Prevention:** Always validate user-supplied file paths using `os.path.commonpath([base_dir, target_path]) == base_dir` to ensure they don't escape their designated parent directories. For unhandled exceptions, always log the error with `exc_info=True` and raise a generic 500 HTTPException.
